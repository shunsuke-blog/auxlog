# 要件定義書 - 筋トレメニュー自動提案アプリ

## 1. プロダクト概要

### プロダクト名
Calcul

### コンセプト
**「今日のあなたの筋トレメニューを、30秒で提案する」**

中級者・上級者向けの筋トレ記録・メニュー自動提案Webアプリ。
種目は自分で決めている。でも今日何をどれだけやるべきかの判断が面倒。
その判断をアプリに任せることで、朝の判断疲れをゼロにする。

### ターゲットユーザー
- 筋トレ歴1年以上の中級者・上級者
- やる種目はある程度決まっているが、セット設計で伸び悩んでいる or 面倒と感じている人
- ジムに週2〜5回通っている社会人

### 解決するペイン
- ジムに行ってから「今日何をやるか」を考えるのが面倒
- 感覚に頼るとベンチプレスなど好きな種目に偏る
- 前回の記録を覚えておらず、重量・回数の増減判断ができない

## 2. 機能要件

### MVP機能（フェーズ1）

#### 2.1 種目登録・編集
- システム側があらかじめ一般的な種目マスタを保持している
- ユーザーはマスタの中から「自分がやる種目」を選ぶだけ（タップで完了）
- 種目名・部位の手入力は不要
- マスタにない種目は独自種目として追加できる（その場合のみ名前・部位を入力）
- 選んだ種目ごとに以下を設定できる
  - 標準セット数
  - 標準レップ数
  - 表示順（ドラッグで並び替え）

**システム種目マスタ（初期登録済み）**
| 種目名 | 対象部位 |
|--------|---------|
| ベンチプレス | 胸 |
| インクラインベンチプレス | 胸 |
| ダンベルフライ | 胸 |
| スクワット | 脚 |
| レッグプレス | 脚 |
| ルーマニアンデッドリフト | 脚 |
| デッドリフト | 背中 |
| チンアップ | 背中 |
| ラットプルダウン | 背中 |
| ベントオーバーロウ | 背中 |
| オーバーヘッドプレス | 肩 |
| サイドレイズ | 肩 |
| バーベルカール | 腕 |
| トライセプスプレスダウン | 腕 |

#### 2.2 トレーニング記録の入力
- トレーニング実施日
- 実施した種目
- 各種目の重量・セット数・レップ数
- 主観的疲労度（1〜5のスケール）
- メモ（任意）

#### 2.3 次回メニューの自動提案
以下のルールベースロジックで提案する（MVPはAI不要）
根拠：筋肥大研究の最新メタアナリシス（Schoenfeld et al. 2021、RP Strength 2025、PubMed 2025）に基づく

**種目選択ロジック**
- 前回トレーニングからの経過日数が長い種目を優先
- 前回トレーニングから48時間未満の部位は後回し（筋タンパク質合成の回復を考慮）
- 同日に同じ部位を連続しないよう考慮
- ユーザーが登録している種目の中からのみ提案（知らない種目は出さない）

**重量・セット提案ロジック**
プログレッシブオーバーロードの優先順位に従い、以下の順で判定する

| 状況 | 提案 |
|------|------|
| 全セットでRIR2以上（余裕あり）を達成 | 重量+2.5kg |
| 一部のセットでRIR2未満（ギリギリ） | 同重量・同レップ維持 |
| レップ数が目標に届かなかった | 重量据え置き、レップ目標を1下げる |
| 3週間以上重量・レップが停滞 | セット数+1を提案 |
| 疲労度4以上 | 重量を5%減、回復優先セッションを提案 |

**週ボリューム管理ロジック**
- 各筋群の週セット数を自動集計して表示
- 最適範囲：週10〜20セット/筋群（中級者・上級者の筋肥大に有効な範囲）
- 週セット数が10未満 → セット追加を提案
- 週セット数が20超 → オーバートレーニングの警告を表示

**RIRの入力方法**
- 各セットの余裕度をトグルで入力（2択）
  - 🔴 限界（RIR0〜1：あと1回もできない状態）
  - 🟢 余裕あり（RIR2以上：まだ余裕がある状態）
- デフォルトは「余裕あり」
- タップ1回で切り替え

#### 2.4 トレーニング履歴の閲覧
- 種目別の重量推移グラフ
- 過去のトレーニングログ一覧
- 総トレーニングボリューム（重量×レップ数×セット数）の推移

### フェーズ2以降（MVP後）

- AIによる提案精度向上（claude-haiku使用）
- 疲労度の自然言語入力（「今日は肩が重い」など）
- 英語対応
- ネイティブアプリ化（iOS/Android）
- プッシュ通知（トレーニング日のリマインド）

## 3. デザイン原則

### 絶対に守るルール
このアプリのコアバリューは「開いた瞬間に今日のメニューがわかる」だ。機能追加の判断は常にこの原則に照らして行う。

**追加前に必ず問う3つの質問**
1. この機能はコアバリューを強化するか？
2. この機能がないと誰かがアプリをやめるか？
3. この機能を追加することで、他の機能が使いにくくなるか？

3つ目がYESなら追加しない。1と2がどちらもNOなら追加しない。

**具体的な制約**
- ホーム画面に表示する情報は「今日やる種目・重量・セット数」のみ
- 1回の記録入力は30秒以内に完了できること
- ナビゲーションは最大4項目まで
- 新機能はフェーズ2以降に持ち越す。MVPに追加しない

### 参考にする思想
- 「引き算のデザイン」：迷ったら削る
- 競合（GymworkやGYMLOG AI）が機能を増やしてきたとしても、追いかけない

### UI/UXデザイン方針
**目指す世界観：シンプル・スタイリッシュ・洗練**
- 筋トレに真剣に向き合う人間が使うツールとして、安っぽさを排除する
- 余白を意識的に使い、情報密度を下げる
- フォント・カラー・アイコンに一貫性を持たせる

**具体的な指針**
- カラーパレットは3色以内に絞る（メインカラー・背景・アクセント）
- フォントは1〜2種類まで
- アイコンはlucide-reactで統一
- ボタンやカードの角丸・余白・影を統一する
- アニメーションは最小限（画面遷移・保存完了のフィードバックのみ）
- テキストは太字・サイズの強弱で階層を作り、色で区別しない

**判断基準**
「Appleの純正アプリと並べて恥ずかしくないか」を常に問う

## 4. 非機能要件

### パフォーマンス
- トップページ（メニュー提案画面）のロードは3秒以内
- 記録入力から保存完了まで1秒以内

### セキュリティ
- Supabase AuthによるGoogle/メール認証
- RLS（Row Level Security）でユーザーデータを完全分離
- HTTPS必須

### レスポンシブ対応
- スマートフォンファーストで設計（ジムでスマホから使う想定）
- PC・タブレットでも使用可能

## 6. 画面設計

### 画面一覧
1. **ホーム（メニュー提案画面）** - アプリを開いたら即座に今日のメニューが表示される
2. **記録入力画面** - トレーニング後に結果を入力
3. **履歴画面** - 過去のログと重量推移グラフ
4. **種目管理画面** - 種目の登録・編集・削除
5. **設定画面** - プロフィール・課金管理

### ホーム画面の設計思想
- アプリを開いた瞬間に今日やるべき種目・重量・セット数が表示される
- 余計な操作ゼロ
- 「記録を入力する」ボタンへの導線が明確

## 7. マネタイズ設計

### モデル
サブスクリプション（月額課金）＋30日間無料トライアル

### 料金
¥480/月（税込）

### 無料トライアル
- 期間：30日間
- 全機能が使える
- クレジットカード登録あり（Stripe）
- 30日後に自動課金開始、事前通知メールあり

### トライアル終了後
- 課金継続 → 全機能利用可能
- 解約 → アカウントとデータは保持、記録閲覧のみ可能（新規記録・提案機能は停止）

### 収益目標との整合性
- 月5万円達成に必要な有料ユーザー数：約105人
- トライアル→有料転換率20%を目標とすると、3ヶ月で525人のトライアル登録が必要
