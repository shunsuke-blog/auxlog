# 要件定義書 - 筋トレメニュー自動提案アプリ

## 1. プロダクト概要

### プロダクト名
Auxlog

### コンセプト
**「今日のあなたの筋トレメニューを、30秒で提案する」**

中級者・上級者向けの筋トレ記録・メニュー自動提案Webアプリ。
種目は自分で決めている。でも今日何をどれだけやるべきかの判断が面倒。
その判断をアプリに任せることで、朝の判断疲れをゼロにする。

### ターゲットユーザー
- 筋トレ歴1年以上の中級者・上級者
- やる種目はある程度決まっているが、セット設計で伸び悩んでいる or 面倒と感じている人
- ジムに週2〜5回通っている社会人

### 解決するペイン
- ジムに行ってから「今日何をやるか」を考えるのが面倒
- 感覚に頼るとベンチプレスなど好きな種目に偏る
- 前回の記録を覚えておらず、重量・回数の増減判断ができない

## 2. 機能要件

### フェーズ1（実装済み）

#### 2.1 種目登録・編集
- システム側があらかじめ一般的な種目マスタを保持している
- ユーザーはマスタの中から「自分がやる種目」を選ぶだけ（タップで完了）
- 種目名・部位の手入力は不要
- マスタにない種目は独自種目として追加できる（その場合のみ名前・部位を入力）
- 選んだ種目ごとに以下を設定できる
  - 標準セット数
  - 標準レップ数
  - 表示順（ドラッグで並び替え）
- 自重種目フラグ（`is_bodyweight`）を持つ（チンアップ等の加重オプション表示に使用）
- コンパウンド種目フラグ（`is_compound`）を持つ（回復日数の計算に使用）

**システム種目マスタ（初期登録済み）**
| 種目名 | 対象部位 | 自重 | コンパウンド |
|--------|---------|------|------------|
| ベンチプレス | 胸 | - | ✅ |
| インクラインベンチプレス | 胸 | - | ✅ |
| ダンベルフライ | 胸 | - | - |
| スクワット | 脚 | - | ✅ |
| レッグプレス | 脚 | - | ✅ |
| ルーマニアンデッドリフト | 脚 | - | ✅ |
| デッドリフト | 背中 | - | ✅ |
| チンアップ | 背中 | ✅ | ✅ |
| ラットプルダウン | 背中 | - | ✅ |
| ベントオーバーロウ | 背中 | - | ✅ |
| オーバーヘッドプレス | 肩 | - | ✅ |
| サイドレイズ | 肩 | - | - |
| バーベルカール | 腕 | - | - |
| トライセプスプレスダウン | 腕 | - | - |

#### 2.2 トレーニング記録の入力
- トレーニング実施日（日付変更可能、過去日付に遡って入力可）
- 実施した種目
- 各種目の重量・セット数・レップ数
- ウォームアップフラグ（W）: セットをウォームアップとしてマーク
- 実施フラグ（done）: セットを実際に行ったかのチェック。ONのセットのみ保存
- 主観的余裕度（RIR）: 限界 / 余裕あり の2択トグル（ウォームアップ時は非表示）
- 主観的疲労度（1〜5のスケール）
- メモ（任意）
- 自重種目は加重をオプション表示（「+ 加重」ボタンで表示切替）
- RIRのデフォルトは「限界」（false）

#### 2.3 次回メニューの自動提案
以下のルールベースロジックで提案する（MVPはAI不要）

**トレーニングレベル**
ユーザーのトレーニング歴に応じて週ボリューム目標を個人化する。

| レベル | 目安 | 週目標セット数（筋群ごと） |
|--------|------|--------------------------|
| 初級者 | 1年未満 | 8〜12セット |
| 中級者 | 1〜3年 | 12〜16セット |
| 上級者 | 3年以上 | 16〜20セット |
根拠：筋肥大研究の最新メタアナリシス（Schoenfeld et al. 2021、RP Strength 2025、PubMed 2025）に基づく

**種目選択ロジック**
- 前回トレーニングからの経過日数が長い種目を優先（降順ソート）
- 前回のセット内容・種目タイプに応じた最低回復日数を下回る種目は除外
  - コンパウンド種目 × 限界セットあり → 3日間除外
  - アイソレーション種目 × 限界セットあり → 2日間除外
  - 全セット余裕あり（RIR=true）→ 2日間除外
  - 初回（前回記録なし） → 常に提案
- 7日以上経過した種目は「優先」としてアクセントカラーボーダーで視覚的に強調
- ユーザーが登録している種目の中からのみ提案

**重量・セット提案ロジック**
プログレッシブオーバーロードの優先順位に従い、以下の順で判定する

| 状況 | 提案 |
|------|------|
| 初回（記録なし） | default_sets × default_reps の直線セット |
| 疲労度 >= 4（有酸素種目） | 重量 × 0.95（2.5kg丸め）、レップ数維持 |
| 疲労度 >= 4（自重種目） | レップ数 × 0.8（20%減） |
| トップセットがレップ未達 | 重量維持、目標レップを前回+1回 |
| 全トップセットが余裕あり・レップ達成（有酸素） | 重量 +2.5kg、default_reps でリセット |
| 全トップセットが余裕あり・レップ達成（自重） | レップ数 +2回 |
| 3セッション停滞（ストール）| セット数 +1 |
| ギリギリ達成 | 重量・レップ数維持 |

**判定の基準**
- 判定にはウォームアップセットを除外したワーキングセットのみ使用
- トップセット（最大重量のセット）のみでRIR・レップ判定を行う
- 前回の各セット重量パターンを引き継ぐ（ピラミッド・逆ピラミッド対応）
- ストール判定はレップ達成時のみ適用（未達は進捗中のため除外）

**週ボリューム管理ロジック**
- 各筋群の週セット数を自動集計して表示
- 最適範囲：週10〜20セット/筋群（中級者・上級者の筋肥大に有効な範囲）
- 週セット数が10未満 → ボリューム不足インジケーターを表示
- 週セット数が20超 → オーバートレーニング注意インジケーターを表示

**RIRの入力方法**
- 各セットの余裕度をトグルで入力（2択）
  - 限界（RIR0〜1：あと1回もできない状態）→ 赤系
  - 余裕あり（RIR2以上：まだ余裕がある状態）→ 緑系
- デフォルトは「限界」（false）
- タップ1回で切り替え
- ウォームアップセットはRIRトグルを非表示

#### 2.4 ホーム画面
- アプリを開いた瞬間に今日のメニューが提案表示される（SSRによる高速表示）
- 各提案カードをスワイプ左で「今日はやらない」として非表示にできる
- 「+」ボタンで非表示にした種目や追加したい種目をその日のメニューに追加できる
- 非表示状態はsessionStorageで当日のみ保持（ページリロード後も維持）
- 全種目を完了させると「今日のメニュー完了！」メッセージを表示

#### 2.5 トレーニング履歴の閲覧
- 週次カレンダー（WeekCalendar）で日付を選択してセッション絞り込み
- 月次カレンダーモーダル（MonthCalendar）でより遠い日付に移動可能
- 種目ごとの重量推移グラフ（最大重量・総挙上量・推定1RM の3メトリクス切り替え）
- 過去のトレーニングログ一覧（日付・疲労度・種目・セット詳細）
- 同一日に複数セッションが存在する場合はまとめて1件として表示・編集
- セッション全体の編集ボタン（`/record/edit/[sessionId]`）
- 種目ごとの編集ボタン（`/record/edit/[sessionId]?exerciseId=[id]`）
- セッションの削除機能
- 自重種目のボリュームは回数（rep数）で表示
- 有酸素種目のボリュームはkg（重量×回数）で表示

#### 2.6 記録編集
- 過去のトレーニングセッションを編集できる（`/record/edit/[sessionId]`）
- 種目を絞り込んで特定種目のみ編集できる（`?exerciseId=[id]` クエリ）
- 全体編集モードでは種目の有効/無効トグル（CircleCheck）・種目追加モーダルが使用可能
- 同一日に複数セッションがある場合は `?merge=id1,id2` でまとめて編集し保存時に統合
- セッション削除機能（確認ダイアログあり）
- RPC関数（`update_session_with_sets`）によるアトミック更新

### フェーズ2以降（MVP後）

## 3. デザイン原則

### 絶対に守るルール
このアプリのコアバリューは「開いた瞬間に今日のメニューがわかる」だ。機能追加の判断は常にこの原則に照らして行う。

**追加前に必ず問う3つの質問**
1. この機能はコアバリューを強化するか？
2. この機能がないと誰かがアプリをやめるか？
3. この機能を追加することで、他の機能が使いにくくなるか？

3つ目がYESなら追加しない。1と2がどちらもNOなら追加しない。

**具体的な制約**
- ホーム画面に表示する情報は「今日やる種目・重量・セット数」のみ
- 1回の記録入力は30秒以内に完了できること
- ナビゲーションは最大4項目まで
- 新機能はフェーズ2以降に持ち越す。MVPに追加しない

### 参考にする思想
- 「引き算のデザイン」：迷ったら削る
- 競合（GymworkやGYMLOG AI）が機能を増やしてきたとしても、追いかけない

### UI/UXデザイン方針
**目指す世界観：シンプル・スタイリッシュ・洗練**
- 筋トレに真剣に向き合う人間が使うツールとして、安っぽさを排除する
- 余白を意識的に使い、情報密度を下げる
- フォント・カラー・アイコンに一貫性を持たせる

**カラーシステム**
- ライトモード: 背景 `#ffffff`、テキスト `#000000`、アクセント `#B8CC00`
- ダークモード: 背景 `#0A0A0A`、カード `#141414`（zinc-950相当）、アクセント `#E8FF00`
- アクセントカラーは優先バッジ・強調ボーダーなど限定的に使用

**具体的な指針**
- カラーパレットは3色以内に絞る（メインカラー・背景・アクセント）
- フォントは1〜2種類まで
- アイコンはlucide-reactで統一
- ボタンやカードの角丸・余白・影を統一する（`rounded-3xl`、`shadow-[0_2px_16px...]`）
- アニメーションは最小限（画面遷移・保存完了のフィードバックのみ）
- テキストは太字・サイズの強弱で階層を作り、色で区別しない
- ヘッダーは全画面で `sticky top-0` 固定、高さ統一（`py-5`）

**PWA対応**
- `viewport-fit: cover` による全画面対応
- `env(safe-area-inset-bottom)` でBottomNav・モーダル・固定ボタンをiOSノッチ対応

**判断基準**
「Appleの純正アプリと並べて恥ずかしくないか」を常に問う

## 4. 非機能要件

### パフォーマンス
- トップページ（メニュー提案画面）のロードは3秒以内
- 記録入力から保存完了まで1秒以内

### セキュリティ
- Supabase AuthによるGoogle OAuth認証（メール認証は廃止）
- RLS（Row Level Security）でユーザーデータを完全分離
- HTTPS必須
- 全APIにzodによるバリデーションを適用
- APIエラーメッセージは汎用化（内部情報の漏洩を防止）
- セッション取得のlimitはサーバー側で最大100件に強制

### レスポンシブ対応
- スマートフォンファーストで設計（ジムでスマホから使う想定）
- PC・タブレットでも使用可能

## 8. マネタイズ設計

### モデル
サブスクリプション（月額課金）＋30日間無料トライアル

### 料金
¥480/月（税込）

### 無料トライアル
- 期間：30日間
- 全機能が使える
- クレジットカード登録不要でトライアル開始（Stripe Subscription: trial_period_days=30）
- 30日後に自動課金開始（要カード登録）、事前通知メールあり（Stripe trial_will_end webhook）

### サブスクリプションステータス
| ステータス | 意味 | アクセス |
|-----------|------|---------|
| `trialing` | トライアル期間中 | 全機能 |
| `active` | 有効（課金中） | 全機能 |
| `canceling` | 解約予定（期間終了まで有効） | 全機能（trial_ends_at まで） |
| `canceled` | 解約済み | 記録閲覧のみ |
| `past_due` | 支払い遅延 | 記録閲覧のみ |

### トライアル終了後
- 課金継続 → 全機能利用可能
- 解約 → アカウントとデータは保持、記録閲覧のみ可能（新規記録・提案機能は停止）
- 再開 → `/subscribe` ページからStripe Customer Portal経由でカード登録・再契約可能

### 管理者アカウント
- `is_admin = true` のユーザーはサブスクリプションチェックをスキップ
- 課金なしで全機能が常時利用可能

### 無料ユーザーアカウント
- `is_free = true` のユーザー（ベータテスター・招待ユーザー等）はサブスクリプション不要で全機能利用可能
- `free_until = NULL` → 永久無料
- `free_until = 日時` → 期限付き無料（期限切れ後は通常のサブスクチェックへ）
- 設定画面にステータスを「無料」または「無料（期限付き）」で表示
- Supabase の users テーブルで直接 `is_free` / `free_until` を設定して管理

### 収益目標との整合性
- 月5万円達成に必要な有料ユーザー数：約105人
- トライアル→有料転換率20%を目標とすると、3ヶ月で525人のトライアル登録が必要
